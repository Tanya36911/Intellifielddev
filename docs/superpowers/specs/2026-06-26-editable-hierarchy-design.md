# Editable Hierarchy Design Spec (Setup Wizard, Slice 1)

**Date:** 2026-06-26
**Screen:** Admin Hierarchy (`/hierarchy`) gains an Edit mode
**Status:** Approved by Tanya (delegated: "you review and do it yourself"). Scope settled in brainstorm.

## Context

This is slice 1 of 2 toward the setup wizard. The wizard needs to add, rename, and
delete org nodes, which the backend cannot do yet (the tree is read-only). This
slice builds that node-editing brick and turns the existing read-only Hierarchy
screen (W7) into an editable one. Slice 2 (later) is the 5-step wizard UI, which
will also add org-level editing. Decisions from the brainstorm:

- Operations in scope: **add, rename, delete** a node. **Move/re-parent is deferred.**
- **Org-level editing is deferred** to the wizard slice (Lumen already has its levels).
- **Delete is allowed only when the node is empty** (no children, no pinned users,
  no assigned surveys, no responses); otherwise it is refused with a clear reason.
- **Codes are auto-generated** from the name (a non-coder never types a slug).
- **Add is contextual** (you add a child under a node; its level is parent + 1).

## Backend brick (MAIN folder, test-first; no migration)

The `nodes` table already exists: `id, tenant_id, parent_id, level_order, name,
code, path, chain, address, lat, lng, tz, created_at`, unique `(tenant_id, code)`,
path is a materialized `/<id>/<id>/.../` string. All three endpoints are admin-only
(`require_admin`) and branch-scoped through the `ScopedRepo`.

### POST /nodes  (admin)
Body: `{ parent_id, name, chain?, address?, lat?, lng?, tz? }`.
- The parent must be in the caller's tenant and scope (else 404).
- The new node's `level_order` is the parent's `+ 1`. Adding a child under a node
  already at the locked bottom level (a Store) is refused (400 "cannot add below
  the lowest level"), since a store is a leaf.
- `code` is auto-generated by slugifying `name` (lowercase, non-alphanumerics to
  hyphens) and, if that collides within the tenant, appending `-2`, `-3`, ... .
- Insert, then set `path = parent.path || new_id || '/'` (same two-step the seed uses).
- Store attributes (chain/address/lat/lng/tz) are accepted on any level but are
  only meaningful at the Store level; the frontend only collects them there.
- Returns the created node in the `GET /nodes` row shape (200).

### PATCH /nodes/{id}  (admin)
Body (all optional, at least one): `{ name?, chain?, address?, lat?, lng?, tz? }`.
- `parent_id`, `level_order`, `path`, and `code` are NOT editable (structure +
  the permanent id stay put; renaming changes the display name, not the code).
- The node must be in the caller's tenant and scope (else 404). Returns the row.

### DELETE /nodes/{id}  (admin)
- The node must be in the caller's tenant and scope (else 404).
- Refused with 409 and a specific reason if it is not empty: it has child nodes,
  or users pinned to it (`assignments`), or survey assignments
  (`survey_assignments`), or responses (`responses`). The message names the first
  blocker found.
- If empty, the row is deleted. Returns `{ "ok": true }` (200).

### ScopedRepo additions
`create_node(parent_id, name, attrs) -> dict | None` (None if parent out of scope),
raising a `ValueError("bottom")` when adding under the locked level; `update_node(
node_id, fields) -> dict | None`; `delete_node(node_id) -> str | None` returning a
blocker reason string when not empty, None on success, and a sentinel for
out-of-scope (the router maps these to 404 / 409 / 200). Plus a `_slug_code(conn,
name)` helper. Reuse the existing `_max_level(conn)`.

### Tests (api/tests/test_nodes_write.py)
admin adds a child (level = parent+1, code auto-generated, path correct); add under
a store refused (400); non-admin add/patch/delete refused (403); add with an
out-of-scope parent 404; rename changes name not code; patch out-of-scope 404;
delete an empty leaf works; delete refused when it has children / a pinned user /
a survey assignment / responses (409 with reason); company isolation (cannot touch
another tenant's node); auto-code uniqueness (two nodes named the same get distinct
codes).

## Frontend (editable Hierarchy screen)

`apps/admin/src/pages/Hierarchy/` gains:
- An **Edit mode** toggle in the Topbar (admin only; replaces the disabled "soon"
  Edit button). Managers and reps never see it; the screen stays read-only for them.
- In edit mode, each tree row shows row actions: **add child** (+), **rename**,
  **delete** (a small icon cluster, revealed in edit mode only). The bottom
  (Store) level shows no add-child action.
- A `NodeFormModal` (add + rename) with a name field, and chain + address fields
  shown only when the node being added/edited is at the Store level. On add it
  shows the parent and the resulting level ("New District under West"). Surfaces
  backend errors inline.
- Delete uses a small confirm; if the backend refuses (409), the modal/inline
  message shows the returned reason ("Has 3 child nodes", etc.).
- New hook code in `useHierarchy.ts`: `useCreateNode`, `useUpdateNode`,
  `useDeleteNode` mutations (via `apiSend` for POST/PATCH; a new `apiDelete` helper
  in `lib/api.ts` for DELETE), all invalidating `['nodes']`.

Role-gating: edit controls only when `session.user.role === 'admin'`; the backend
is the real guard. Pure helpers (slug preview, the add-context label) are unit
tested; the screen has an edit-mode integration test (add opens the modal and
POSTs; non-admin sees no Edit toggle; delete calls DELETE).
