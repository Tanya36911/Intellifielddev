## Implementation Notes

### Edit-published -> new-version flow

When `useSurvey(id)` returns a survey where all versions have `published_at !== null` (all published, no draft), the Builder shows an amber banner: "Editing creates a new version." On the first save, it calls `useNewVersion(id)` to spin a fresh draft. A 409 response is handled by refetching and using the existing draft instead.

### Publish -> assign navigation

The publish flow:
1. `saveDraft()` is called first to ensure the survey exists and has a draft.
2. The `PublishConfirm` modal is shown.
3. On confirm, `publish.mutateAsync(surveyId)` is called.
4. The returned `SurveyDetail.versions` is filtered for entries where `published_at !== null`, then the one with the highest `version_number` is selected as the published version.
5. `navigate('/surveys/' + surveyId + '/assign', { state: { versionId: best?.id, name } })` is called.
6. A 409 publish-race shows "Already published; reloading" and calls `refetch()`.

### Reorder test

The second test adds a "Yes / No" question (prompt "First question") and a "Number" question (prompt "Second question"), then clicks "Move down" on the first question (index 0). After the swap, `getAllByPlaceholderText(/question/i)` shows "Second question" first and "First question" second.

- The `adminSession` fixture is a function (`adminSession()`), not a plain object - used consistently with `()` in all tests.
- The test for "first save" asserts `send` called with `'POST', '/surveys', ...` and `body.questions[0].type === 'boolean'` -- the Builder correctly maps `blankQuestion('boolean')` through `mapToBackendQuestion` which passes through `type: 'boolean'`.
